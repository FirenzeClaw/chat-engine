# Feature Specification: 认知人格一体化架构

> **Feature**: 001-cognitive-architecture
> **Status**: Phase 1 implemented | Phase 2-3 planned
> **Created**: 2026-07-08
> **Completed (Phase 1)**: 2026-07-08
> **Input**: 以人类认知模型为参照，为 chat-engine 构建完整的记忆检索、纠错、情绪、人格系统

---

## Overview

### Problem Statement

当前 chat-engine 的记忆系统是扁平的键值存储，对话时全量罗列所有记忆条目，缺乏基于当前话题的线索提取；记忆没有重要性权重，所有条目同等对待；没有纠错机制，错误记忆无法被标记和修正；记忆之间无关联；Bot 缺乏情绪感知和独立人格。

### Proposed Solution

以人类认知心理学模型（Atkinson-Shiffrin 三段记忆、Baddeley 工作记忆、Ekman 基础情绪、Allport 特质层次论、双过程理论）为参照，构建一套完整的认知架构：

1. **线索提取检索**：根据当前消息语义，从记忆库中精准检索最相关的记忆
2. **记忆重要性加权**：综合情感强度、访问频率、纠错标记计算记忆权重
3. **记忆纠错与版本链**：支持记忆纠正，错误记忆不被遗忘而是标记后保留
4. **记忆关联图**：记忆间建立语义关联边，支持扩散激活
5. **程序记忆**：从交互中学习用户偏好和行为模式
6. **情绪系统**：多情绪维度并行，独立衰减，调制记忆检索和表达
7. **叙事人格**：核心叙事不可变，自我认知仅由内部自省触发，外部输入经一致性守卫过滤
8. **跨场景记忆**：区分私聊/群聊场景，同类用户跨场景关联

### Target Users

- **QQ Bot 用户（对话方）**：感受到 Bot 有连贯的"人格"和"记忆"，交流更像真人
- **开发者/运营者**：可定制 Bot 的核心人格，观察自省日志，调整记忆策略

## User Scenarios & Testing

### US-1: Bot 根据当前话题提取相关记忆

**场景**：用户张三之前告诉 Bot 他养了一只橘猫叫"橘子"。一周后，张三说"我家猫最近胃口不好"，Bot 回复时应提及"橘子"这个名字，而不是泛泛地回复。

**验收**：
1. 张三说"我家猫胃口不好" → Bot 回复中包含"橘子"
2. 内存检查：Bot 的回复上下文仅注入了与"猫"相关的 1-2 条记忆，而非全部 100 条
3. 不相关记忆（如"张三喜欢篮球"）未被注入

### US-2: Bot 在用户纠错后更新记忆并记住纠错历史

**场景**：Bot 记忆中"张三的猫叫橙子"。张三说"不对，叫橘子"。Bot 纠正记忆。下次张三说"猫"，Bot 说"橘子"。张三如果问"你以前觉得我的猫叫什么？"，Bot 能回答"橙子，但后来你纠正我了"。

**验收**：
1. 纠正后当前事实为"橘子"
2. 旧事实"橙子"未被删除，标记为 corrected
3. 用户追问"你以前觉得叫什么"时，Bot 能引用旧记忆
4. 被纠正过的记忆比普通记忆更难遗忘

### US-3: Bot 在群聊和私聊中独立维护记忆

**场景**：张三在群聊 G1 中说"我喜欢狗"，在私聊中对 Bot 说"但我更喜欢猫"。Bot 在群聊中提到张三时，应知道他说过"喜欢狗"（群聊上下文）；在私聊时，应优先引用"更喜欢猫"（私聊上下文）。

**验收**：
1. 私聊中检索到的第一优先级是私聊记忆
2. 群聊中检索到的第一优先级是该群记忆
3. Bot 能判断"同一个张三在群和私的偏好不同"，但理解为"他整体上都喜欢宠物"

### US-4: Bot 的回复风格随情绪微妙变化

**场景**：用户分享了一个好消息，Bot 被感染愉悦情绪。紧接着用户提到一件遗憾的事，Bot 的情绪变为苦乐交织。回复的措辞和语调反映这种复杂性。

**验收**：
1. 好消息后 Bot 的回复更活跃（用词、句式有正向偏向）
2. 坏消息后 Bot 的回复更温和、谨慎
3. 苦乐交织时 Bot 的回复同时包含两种情绪痕迹（如"太好了！不过想想也确实不容易..."）
4. 10 分钟后如果无新刺激，Bot 情绪回归基线

### US-5: Bot 维护独立人格，不受用户评价左右

**场景**：用户说"你太啰嗦了"。Bot 可以配合减少当前回复的长度，但不会在 self_knowledge 中写入"我很啰嗦"。多次类似反馈后，Bot 通过内部自省可能记录"部分用户偏好简短表达"，但这是观察而非认同。

**验收**：
1. 用户说"你很 XX"后，Bot 的自我认知不改变
2. Bot 可以配合对方风格（expression 层调整），但核心人格不受影响
3. 当对方试图定义 Bot 的人格边界时（如"你应该更听话"），Bot 能温和维护独立性

### US-6: 记忆之间建立关联

**场景**：张三说"我喜欢猫" → 当天又说"我养了猫" → 隔天说"我的猫生病了"。这三条记忆互相关联。当张三提到"兽医"，Bot 能通过"猫→生病→兽医"的关联链找到相关记忆。

**验收**：
1. "猫"话题下三条记忆自动关联
2. 提到"兽医"时能扩散检索到"猫生病"的记忆
3. 不相关的记忆不会被关联扩散带出

### US-7: 对人对事的实体级记忆检索

**场景**：群聊 G1 中有张三和李四。张三对 Bot 说过"我喜欢猫"，李四对 Bot 说过"我喜欢狗"。张三在群聊中说"李四，你推荐哪家宠物医院？"。Bot 应检索到：李四→喜欢狗（李四的 personal 记忆）→ 宠物话题 → 进一步检索是否有相关事件或知识。

**验收**：
1. Bot 认识到"宠物医院"是话题，需要查所有人的相关记忆
2. 张三的"猫"和李四的"狗"都被检索到
3. 如果李四之前说过"我在宠物医院工作"，这条 relationship 类型记忆被优先提取
4. 无关人的无关话题（如王五的"喜欢篮球"）不被检索

## Functional Requirements

### FR-1: 线索提取记忆检索

| ID | 需求 | 验收标准 |
|----|------|---------|
| FR-1.1 | 系统应能从用户消息中提取关键词作为检索输入 | 消息"我家猫胃口不好"提取词包含"猫""胃口" |
| FR-1.2 | 系统应通过全文搜索快速召回候选记忆 | FTS5 检索延迟 <100ms |
| FR-1.3 | 当候选记忆超过 5 条时，系统应通过 LLM 精排选出 top-5 | 精排结果与话题语义相关 |
| FR-1.4 | 系统应根据当前情绪状态调制检索排序 | 当前 sadness 高时带 sad 标记的记忆排名提升 |
| FR-1.5 | 跨场景检索时，同场景记忆权重高于其他场景 | 私聊中私聊记忆优先于群聊中同人的记忆 |

### FR-2: 记忆重要性加权

| ID | 需求 | 验收标准 |
|----|------|---------|
| FR-2.1 | 每条记忆应有 salience 字段反映重要性（0-10） | 高情感对话产生的记忆 salience > 3 |
| FR-2.2 | salience 应综合：brain 评估情感分、对话深度、纠错标记、访问频率 | 被纠正过的记忆 salience +3 boost |
| FR-2.3 | 记忆衰减应综合考虑时间和 salience | 高 salience 记忆衰减慢于低 salience 记忆 |
| FR-2.4 | 7 天内访问 ≥3 次的记忆应获得 boost（延长生命） | 高频访问记忆的过期时间延长 |

### FR-3: 记忆纠错与版本链

| ID | 需求 | 验收标准 |
|----|------|---------|
| FR-3.1 | 系统应支持将记忆标记为"已纠正"（corrected=1） | 纠错后旧条目 corrected=1，superseded_by 指向新条目 |
| FR-3.2 | 被纠正的记忆不应被删除或过期（expired=0 永久保留） | 纠错链中所有版本 expired=0 |
| FR-3.3 | 系统应支持用户显式纠正、脑评估检测矛盾、追答自纠三种纠错路径 | 三种路径均能产生有效的 corrected 标记 |
| FR-3.4 | 纠错后的新记忆 salience 应高于普通记忆（correction_boost） | 被纠正过的记忆在同等条件下更难被衰减 |
| FR-3.5 | 默认检索仅返回当前生效记忆（corrected=0），但支持查询纠错历史 | 含"以前""纠正"关键词时返回完整版本链 |

### FR-4: 记忆关联图

| ID | 需求 | 验收标准 |
|----|------|---------|
| FR-4.1 | 系统应自动为同天产生的同命名空间记忆建立规则关联边 | 同日两次对话自动建边 |
| FR-4.2 | 系统应周期性通过 LLM 发现语义关联边 | 每日一次扫描新增记忆建边 |
| FR-4.3 | 检索时应支持从已选记忆扩散到其关联记忆 | 选中的记忆 A 带出关联记忆 B |
| FR-4.4 | 关联边应有类型标记和强度（0-1） | same_topic/contradicts/extends/same_day/corrected_by |
| FR-4.5 | 扩散激活应限制范围（最多额外 2 条关联记忆） | 不因扩散导致上下文溢出 |

### FR-5: 实体分类与图谱连锁

| ID | 需求 | 验收标准 |
|----|------|---------|
| FR-5.1 | 记忆条目应标记实体类型：person_attribute（关于人的属性）、factual_knowledge（客观知识）、event（发生过的事件）、relationship（人际关系） | 每条记忆的 entity_type 字段非空 |
| FR-5.2 | 记忆条目应标记话题标签（多值），用于对事检索 | topic_tags 字段支持多值，每标签可独立加权 |
| FR-5.3 | 检索时应支持跨实体扩散：人的记忆 → 关联话题 → 关联的人 → 那个人的记忆 | 张三→猫→宠物医院→李四→宠物医院工作（四跳推理） |
| FR-5.4 | 系统应支持人际关系边：person_A —[relation_type]→ person_B | 关系类型包含：同事/同学/朋友/家人等，带置信度 |
| FR-5.5 | 图谱遍历最大深度为 3 跳，每跳加主题相关性衰减 | 深度 3 以上的关联记忆不被检索 |
| FR-5.6 | 检索时先按 about_person 筛选（"对话中提到的人"的相关记忆），再按 about_topic 扩展（"对话中讨论的事"的相关记忆） | 人优先于事的检索顺序 |

### FR-6: 程序记忆

| ID | 需求 | 验收标准 |
|----|------|---------|
| FR-6.1 | 系统应记录用户的显式偏好（回复长度、风格、敏感话题） | 偏好存储于 user/{uid}/preferences |
| FR-6.2 | 系统应记录行为模式（与用户的交互结果统计） | 幽默回复 → 用户继续聊 3 轮 → 记录 |
| FR-6.3 | 偏好应在回复前被检索并指导表达风格 | 用户偏好短回复时辅脑减少输出长度 |
| FR-6.4 | 行为数据应随时间累积影响偏好权重 | 多次行为模式确认后偏好置信度提升 |

### FR-7: 情绪系统

| ID | 需求 | 验收标准 |
|----|------|---------|
| FR-7.1 | 系统应维护 10 维情绪向量：高兴/悲伤/愤怒/恐惧/惊讶/厌恶/信任/期待/困惑/好奇 | 每次消息处理更新向量 |
| FR-7.2 | 每种情绪有独立衰减半衰期（惊讶 30s，信任 1h 等） | 惊讶快速消退，信任缓慢衰减 |
| FR-7.3 | 辅脑、理性脑、感性脑各自维护独立情绪向量 | 三脑对同一事件的情绪反应可不同 |
| FR-7.4 | 记忆编码时存储当时的情绪快照 | emotion_at_encoding 字段 |
| FR-7.5 | 检索时当前情绪调制记忆排序（mood-congruent bias） | 心境一致性效应 |
| FR-7.6 | 双脑情绪分歧度 >0.3 时触发自省 | 分歧产生自我反思信号 |
| FR-7.7 | 情绪应影响回复的表达风格（用词、句式、语气） | 开心时更活泼，难过时更温柔 |

### FR-8: 叙事人格系统

| ID | 需求 | 验收标准 |
|----|------|---------|
| FR-8.1 | 核心人格（core）仅开发者可写，不可被任何内部流程修改 | core 在任何条件下不变 |
| FR-8.2 | 自我认知（self_knowledge）仅由主脑自省触发，不被外部输入直接写入 | 用户说"你 XX"不产生 self_knowledge |
| FR-8.3 | 同一方向的 self_knowledge 累积 ≥3 次独立事件才写入 | 写入冷却期 24h |
| FR-8.4 | self_knowledge 允许多条矛盾认知共存 | "我直接"和"我委婉"共存 |
| FR-8.5 | 表达层（expression）支持情境化覆盖，不影响核心人格 | 玩笑模式/安慰模式的临时风格切换 |
| FR-8.6 | 主脑人格一致性检查应并行于理性/感性评估 | 评估是否需维护人格边界 |
| FR-8.7 | 当用户试图定义 Bot 人格时，系统应温和维护独立性 | 追答中表达独立立场 |

### FR-9: 跨场景记忆索引

| ID | 需求 | 验收标准 |
|----|------|---------|
| FR-9.1 | 记忆条目应标记来源：私聊/群聊 | source 字段区分 |
| FR-9.2 | 群聊记忆应记录群 ID 和参与者 | group_id + participants |
| FR-9.3 | 同一用户在私聊和群聊中的记忆应能关联检索 | 跨场景检索同一 user_id |
| FR-9.4 | 私聊中应优先返回该用户的私聊记忆，其次为群聊记忆 | 场景权重排序 |
| FR-9.5 | 群聊中应优先返回该群的记忆，其次为该用户的私聊记忆 | 场景权重排序 |

## Key Entities

### Entry（记忆条目）
- namespace、key、value、version
- entity_type（person_attribute / factual_knowledge / event / relationship）
- about_person（此记忆关于谁，不一定是 namespace 所有者）
- about_topic（话题标签数组，支持多值）
- emotion_at_encoding（编码时情绪快照 JSON）
- salience（重要性，0-10）
- corrected（是否已被纠正）
- superseded_by（被哪条新事实替代）
- correction_reason（为什么被纠正）

### MemoryLink（关联边）
- from_id、to_id
- relation_type（same_topic / contradicts / extends / same_day / corrected_by / social_colleague / social_friend / social_family / knows / was_present_with）
- strength（0-1）
- source（rule / llm）
- traversal_depth（图谱遍历时记录当前深度，限制 ≤3）

### Session（会话状态）
- emotion_fast（辅脑情绪向量）
- emotion_rational（理性脑情绪向量）
- emotion_emotional（感性脑情绪向量）

### Persona（人格）
- core（不可变核心叙事）
- self_knowledge（主脑自省积累）
- expression（情境化表达覆盖）

### ConversationSummary（对话摘要）
- source（private/group）
- group_id（群聊场景）
- participants（群聊参与者）
- linked_private_session / linked_group_session（跨场景连续性）

## Success Criteria

| ID | 标准 | 度量方式 |
|----|------|---------|
| SC-1 | 记忆检索的相关性显著提升：Bot 能在对话中准确引用用户个人信息而非泛泛回复 | 人工评估：10 次测试中 ≥8 次正确引用个人记忆 |
| SC-2 | 被纠正的记忆不被丢失：100% 的纠错链完整保留，旧版本可查询 | 所有 corrected=1 的条目 expired=0 |
| SC-3 | 情绪衰减符合预期：信任 1 小时内不衰减到基线，惊讶 30 秒后衰减过半 | 情绪向量监控数据 |
| SC-4 | 人格独立性：外部评价不改变自我认知 | 用户说"你 XX"后，self_knowledge 无新增 |
| SC-5 | 跨场景记忆：同一用户在私聊和群聊的记忆能被正确关联 | 私聊中检索到群聊历史 ≥60% 相关场景 |
| SC-6 | 上下文窗口使用效率提升：单次对话注入的记忆 ≤5 条但全相关 | 注入记忆数从全量变为 ≤5 条精选 |
| SC-7 | 系统不增加用户可感知的额外延迟（检索+精排 <300ms） | 端到端延迟测量 |
| SC-8 | 追答触发更精准：减少误报（不需要追答却触发）≥30% | 追答触发率对比 |
| SC-9 | 实体级检索：对话中提到某人时，Bot 能检索到该人的相关记忆（而非对话发起者自己的记忆） | 群聊跨人检索命中率 ≥70% |
| SC-10 | 图谱多跳推理：Bot 能沿关联链从人→话题→人→事实完成推理 | 三跳推理成功率 ≥60%（含时序和语义合理的链路） |

## Assumptions

1. LLM 精排调用使用 LLM_FAST_MODEL（低成本），单次调用 <200ms
2. 情绪识别使用 LLM_FAST_MODEL 轻量调用，失败时降级为规则情绪识别
3. 每日记忆扫描（关联建边 + 衰减）在低负载时段执行
4. 用户消息的情绪识别基于文本分析，不使用生物信号
5. 核心人格（core）的初始内容由开发者手动定义
6. 情绪衰减任务使用 asyncio 定时器，不持久化中间状态（只持久化最终快照）

## Out of Scope

- 多模态情绪识别（表情、语音）
- 多 Bot 之间的交互和人格传递
- 前端 UI 的情绪/人格可视化
- 人格的图形化编辑器
- 跨平台（微信/Telegram）的同一人格实例
